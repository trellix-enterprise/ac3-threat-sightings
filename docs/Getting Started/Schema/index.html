Schema Overview

This article describes the how the schema is represented in our data model. This allows the practitioner to understand the consumption of threat sightings.

IMPORTANT

It is very important understand the schema of the threat sightings. After read this section, we recommend go to the section called Creating your first sightings to see how to create a new threat sighting.

BEFORE YOU BEGIN

Sometimes, behaviors have specific values that are differents for each intrusion, e.g domains, IPs, arguments, folders, etc... For this reason, we use variables to alert other analysts that the value will be different during an investigation. The format that we use is the following:

dstIp: "#{ip}"dstHost: "#{subdomain}.ddns.net"path: "C:\\Users\\#{user}\\AppData"

Threat Sightings are composed by sections. Therefore, when you are creating a new threat sighting, it's neccesary understand each available section to get the most out of your analysis.

All the Threat Sightings must to have the following sections.

  1. header
  2. threatSightings
  3. behavior
  4. footer

Header Section#

FieldTypeDescriptionRequired
headerobjectThe header section contains context information about the threat sighting‚úÖ

A threat sighting always begins with a header section. The fields that can be represented into this section are the following describred:

FieldTypeDescriptionRequired
sightingReportIduuidContains the uuid of the sighting (uuid version 4). There are several ways to generate this uuid. e.g: https://www.uuidgenerator.net/‚úÖ
statusstringestablishes the status of the sighting. Some values will be: (wip|done|pending approval|testing|draft)‚úÖ
descriptionstringA short description about the threat that is being documented‚úÖ
authorstringAuthors who have created the sighting.‚úÖ
acknowledgement stringAnalysts and/or companies which the sighting was based in. If the sighting is own, this information will be the same that author.‚úÖ
contributorarray stringanalysts who have collaborated by adding or modifying sighting information.‚ùå
tlpstringTraffic Light Protocol (TLP). We based in https://www.cisa.gov/tlp.‚úÖ
threatInformationobjectThis key includes different information about the sighting and the context of this.‚úÖ
threatInformation.adversariesarray stringIf the adversary is known or exists several adversaries, you can fill it with the information. In case you don't know the adversary, you can put n/d (Not Determined)‚ùå
threatInformation.malwarearray stringIf a malware was used during the description of the sighting, you can fill it with the information. In case you don't know the adversary, you can put n/d (Not Determined)‚úÖ
threatInformation.lolbasarray stringLOLBAS are all the binaries and scripts that are currently covered in the official repository: https://lolbas-project.github.io/. You can put all the lolbas used in the sighting. In case you don't know the adversary, you can put n/d (Not Determined)‚úÖ
threatInformation.toolsarray stringThere are also tools used in campaigns that are not malware or lolbas. e.g: Mimikatz is a clear example for this field. We recommend ask yourself.. It is a malware? If not... It is a lolbas? If not... It is a tool!‚úÖ
threatInformation.regionsarray objectRegions and contries affected by the campaign.‚ùå
threatInformation.regions.#{ Region }array stringThe key #{Region} is the region from where the country value is. For example: Western Europe with value Germany.‚ùå
threatInformation.industriesarray stringIf you know the industry of the victim, it is important to complete. This information can help others to focus only in some sightings. The following list will help you https://www.cisa.gov/critical-infrastructure-sectors‚ùå

ThreatSightings Section#

FieldTypeDescriptionRequired
threatSightingsarray objectThe threatSightings section contains all the logic and behaviors of the threat.‚úÖ

A threatSightings section might contains differents sighting sections. Each Sighting section, in general, is related to MITRE Tactic, and into each Sighting can exists different behaviors, normally considered as MITRE Techniques and procedures.

FieldTypeDescriptionRequired
sightingstringThe content of the key is a high level description about the threat described. Now, into this section, is described all the behaviors.‚úÖ
iduuidContains the uuid for this specific sighting section (uuid version 4). There several ways to generate this uuid. e.g: https://www.uuidgenerator.net/‚úÖ
behaviorsarrayThe field has not value. Is declared to describes the each behavior.‚úÖ
behaviors.behaviorstringDescription about the procedure that will be introduced into the behavior.‚úÖ

Behavior object#

This is the most important section, since here the analyst needs to describe all the information about the threat. Here is where the events such as Process Created, File Created, Registry Modified and others event types are introduced with as much information as possible.

Note: Not all the fields are required when you are creating threat sighting.

FieldTypeDescriptionRequired
iduuidContains the uuid for this specific behavior section (uuid version 4). There several ways to generate this uuid. e.g: https://www.uuidgenerator.net/‚úÖ
typestringRefers to the event type. e.g | ProcessCreated | NetworkAccessed | ScheduledTaskChanged | NamedPipeConnected | ImageLoaded | Email | FileCreated | FileModified | FileDeleted | RegValueCreated | RegValueModified | RegValueQueried | RegValueRead | RegKeyCreated | RegKeyDeleted | RegKeyEnumerated | ProcessHollowed | DNSQuery | ServiceChanged | API | FileRead‚úÖ
weaponstringWeapon owner of the behavior described. The weapon has to be part of tools|lolbas|malware|.‚úÖ
notesarray stringThis field has been created so that the analyst who is writing the sighting, can describe as much as possible any interesting information for other analysts.‚ùå
att&ck objectIs declared to introduce inside the tactics and techniques‚ùå

Objects by event types#

Each behavior is represented with information of a specific event type, which is declared in the type field mentioned previously. If you want to represent a file creation event, then you have to declare the key called files. However, if you want to represent a network connection, then you have to declare the key called connections.

The following list is the relation of event type with each key.

Event typeTypeKey
ProcessOpenedarray object accesors
ProcessCreatedarray object processes
ScriptExecutedarray object scripts
NetworkAccessed & DNSQueriedarray object connections
ScheduledTaskChanged & ScheduledTaskCreationarray object scheduledTasks
RegKeyCreated & RegKeyModified & RegKeyDeleted & RegKeyRead & RegValueCreated & RegValueDeleted & RegValueModified & RegValueReadarray object registries
FileCreated & FileModified & FileDeleted & FileRead & FileExecutedarray object files
CodeInjectionarray object injections
DriverLoaded & DriverUnloadedarray object drivers
NamedPipeConnectedarray object pipes

WMIExecutedarray object wmi
ApiInvokedarray object apis
EmailUsedarray object emails

This key is declared when a process is opened to do any action. The fields supported for this key are the following.

FieldTypeDescriptionRequired

targetstringProcess target of the action.‚úÖ

processstringProcess that wants to perform some action with the target process. In general, this field will be similar to from.‚ùå

processes#

This key is declared when a new process is executed. The fields supported for this key are the following.

ATT&CK Section#

FieldTypeDescriptionRequired
att&ckobjectIs declared to introduce inside the tactics and techniques‚ùå

For each behavior, if is possible, is highly recommended introduce all the tactics and techniques that have been used. You can check the matrix here: https://attack.mitre.org/

FieldTypeDescriptionRequired
{tactic_name}array stringAll the tactics availables right now for ATT&CK Mitre. (reconnaissance | resourceDevelopment | initialAccess | execution | persistence | privilegeEscalation | defenseEvasion | credentialAccess | discovery | lateralMovement | collection | commandAndControl | exfiltration | impact)‚ùå
{technique_ID - technique_name}stringAll the techniques availables right now for ATT&CK Mitre. The right form to write it is -> "tech_ID - tech_title". e.g: "T1218.010 - Signed Binary Proxy Execution: Regsvr32"‚ùå

threatHunting Section#

FieldTypeDescriptionRequired
threatHuntingarray objectIs declared to introduce inside the queries for hunt.‚ùå

The threatHunting section contains information about how to do threat hunting in relation to the behaviors. Each query represents the way for do it and relates the behaviorId that applies.

Sigma rules are supported here. There are two ways to add a sigma rule.

  1. Creating a new sigma (sigma rules are added from logsource key to condition key)
  2. Referencing an existing rule

Creating a new sigma#

FieldTypeDescriptionRequired
querystringClear and small description of the query that you want to create.‚úÖ
queryIduuidContains the uuid for this specific query section (uuid version 4). There several ways to generate this uuid. e.g: https://www.uuidgenerator.net/‚úÖ
typestringType of the query you are adding. e.g: sigma.‚úÖ
behaviorIdsarray stringBehavior or behaviors mapped to this hunting query.‚úÖ
logsourceobjectCheck the logsource documentation of Sigma.‚ùå
categoryobjectCheck the logsource documentation of Sigma.‚ùå
productobjectCheck the logsource documentation of Sigma.‚ùå
detectionobjectCheck the detection documentation of Sigma.‚ùå
selectionobjectCheck the detection documentation of Sigma.‚ùå
conditionstringCheck the condition documentation of Sigma.‚ùå

Referencing an existing rule#

When you are using an existing rule, it is important to add it as follows

FieldTypeDescriptionRequired
querystringClear and small description of the query that you want to create.‚úÖ
queryIduuidContains the uuid of the real sigma rule.‚úÖ
typestringType of the query you are adding. e.g: sigma.‚úÖ
linkstringLink to the rule you are referencing.‚úÖ
behaviorIdsarray stringBehavior or behaviors mapped to this hunting query.‚úÖ
